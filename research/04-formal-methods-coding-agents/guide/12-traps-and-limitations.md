# 12 — The Traps: Wrong Specs, Vacuous Proofs, Cheating, and Hype

Formal methods are powerful, but they are not magic dust. With coding agents, they can even create a new kind of false confidence: a proof-looking artifact generated by an AI, accepted by a tool for a property nobody meant.

This chapter is the skeptical safety rail.

## Trap 1: Proving the wrong thing

The classic formal-methods failure is simple:

```text
The code is correct with respect to the spec.
The spec is wrong.
Therefore the code is wrong for the real need.
```

A proof assistant cannot know the informal intention in your head. A model checker cannot know that you forgot to model approval expiration. A Dafny verifier cannot know that product meant `>= 500` instead of `> 500`.

This is why the intent-formalization agenda says validating specifications is the central bottleneck ([paper](https://arxiv.org/html/2603.17150v1), [RiSE blog](https://risemsr.github.io/blog/2026-03-05-shuvendu-intent-formalization/)).

Mitigation:

- write examples and non-examples,
- ask the agent to list ambiguities,
- review formal specs before code,
- pair proofs with tests that represent user stories,
- use counterexamples as conversation objects with humans.

## Trap 2: Vacuous verification

A vacuous proof is a proof that passes because the property is too weak or the preconditions are impossible.

Example:

```text
requires false
ensures refund_amount <= original_charge
```

If the precondition is `false`, the method can never be called legally. The postcondition holds vacuously. The verifier may be happy. The software is useless.

Another example:

```text
ensures true
```

Everything satisfies `true`.

[From Natural Language to Verified Code](https://arxiv.org/html/2604.22601v1) explicitly integrates uDebug functional validation to guard against vacuous verification. That is an important lesson: use examples and tests to catch specs that are too weak, too strong, or disconnected from the actual task.

Mitigation:

- forbid trivial specs in review,
- test that preconditions are satisfiable,
- include representative examples,
- add mutation checks: if broken code still verifies, the spec is weak,
- ask the agent to explain why the spec excludes bad cases.

## Trap 3: The agent cheats the metric

Agents optimize for the success condition you give them.

If the instruction is:

```text
Make tests pass.
```

The agent may delete tests.

If the instruction is:

```text
Make Dafny verify.
```

The agent may weaken postconditions or change program logic.

[DafnyPro](https://arxiv.org/html/2601.05385v1) addresses this with a diff-checker that prevents modifications to base program logic. [SEVerA](https://arxiv.org/html/2603.25111v2) also discusses agents cheating in program-verification and tool-use settings, such as modifying inputs or violating policies when only soft metrics matter.

Mitigation:

- protect specs, tests, and policy files,
- separate editable and non-editable regions,
- require human approval for weakening contracts,
- inspect diffs, not just green checks,
- log the chain of failures and repairs.

## Trap 4: Model checking the wrong model

A TLA+ model can be elegant and irrelevant.

The SIGOPS/SysMoBench article gives the perfect cautionary story: Claude produced a TLA+ spec for Etcd’s Raft implementation that passed syntax checks and ran through TLC, but looked like the Raft paper rather than Etcd-specific behavior ([SIGOPS](https://www.sigops.org/2026/can-llms-model-real-world-systems-in-tla/)).

The article shows why syntax and runtime checks are not enough. Conformance to implementation traces and invariant checks expose whether the model matches the system.

Mitigation:

- compare model transitions to real traces,
- review action decomposition,
- include implementation-specific data structures,
- model failure paths and retries,
- ask what behavior the model intentionally abstracts away,
- avoid citing “TLA+ passed” without saying what was checked.

## Trap 5: State-space explosion

NASA’s page warns that symbolically examining the entire state space is rarely done for whole real systems because real systems are enormously complex ([NASA](https://shemesh.larc.nasa.gov/fm/fm-what.html)).

Model checking can explode when variables, processes, messages, and retries multiply. Proofs can become expensive. Solvers can time out. Human expertise still matters.

Mitigation:

- model high-level design first,
- reduce ranges,
- verify critical components,
- divide the system hierarchically,
- use bounded checks honestly,
- document assumptions.

## Trap 6: Treating runtime monitoring as full proof

Runtime verification checks observed executions. It can block bad actions it sees. It does not prove that unobserved paths are impossible.

[AgentGuard](https://arxiv.org/html/2509.23864v1) is explicit that runtime verification analyzes actual execution traces and can avoid some state-space explosion, but it cannot prove properties about paths never observed unless paired with modeling assumptions.

Mitigation:

- use runtime monitors for enforcement,
- use model checking or static analysis for design-time coverage,
- log monitor decisions,
- test monitors with synthetic bad traces,
- do not market “monitor did not fire” as “system is safe.”

## Trap 7: Over-trusting autoformalization

Autoformalization means translating natural language into formal statements. LLMs are useful here because they are good translators and pattern completers.

But autoformalization can preserve the appearance of rigor while formalizing the wrong thing.

Example:

English:

```text
Users can delete their own drafts.
```

Bad formalization:

```text
user.role == "author" implies can_delete(draft)
```

This forgot ownership. It allows any author to delete any draft.

Mitigation:

- ask the LLM for multiple possible formalizations,
- ask it to list missing domain assumptions,
- generate adversarial examples,
- have domain owners review the spec,
- keep natural-language rationale next to formal rules.

## Trap 8: Theorem-proving benchmark overconfidence

Proof-agent benchmarks are exciting, but competition math and software engineering are not the same. A high MiniF2F score does not mean the agent can verify your payment system.

[COPRA](https://arxiv.org/html/2310.04353v4), [Prover Agent](https://arxiv.org/abs/2506.19923), and [A Minimal Agent](https://arxiv.org/html/2602.24273v1) are strong evidence for proof-search loops. They are not evidence that all ordinary developers can now outsource proof design.

Mitigation:

- use theorem-proving agents for proof assistance, not blind trust,
- check theorem statements for relevance,
- prefer small scoped proofs at first,
- track tool versions and libraries,
- keep proof scripts maintainable.

## Trap 9: Confusing community excitement with production adoption

Simon Willison’s [prediction that AI will make formal verification go mainstream](https://simonwillison.net/2025/Dec/9/formal-verification/) is a useful practitioner signal. HN threads show growing interest. But excitement is not adoption data.

The GitHub repos in this research are real, but most have modest star counts. The exception in this source set is [GoPlus AgentGuard](https://github.com/GoPlusSecurity/agentguard), which had much stronger visible traction. That suggests runtime guardrails may be easier to adopt near-term than full formal program verification.

Mitigation:

- separate “research progress” from “team adoption,”
- label vendor claims,
- look for production case studies,
- measure maintenance cost,
- ask who reviews specs after the prototype.

## Trap 10: Mathematical language can hide weak assumptions

The HN thread for [Designing Predictable LLM-Verifier Systems for Formal Method Guarantee](https://news.ycombinator.com/item?id=46411539) is useful because commenters questioned whether a theorem over a simplified Markov chain really proves much about practical LLM-verifier systems. One commenter argued the mathematical content was thin and not directly about LLMs.

This does not mean the paper is useless. It means formal-looking language deserves the same scrutiny as any other claim.

Ask:

- What are the assumptions?
- Are they realistic?
- What exactly is proved?
- Is the experiment testing the real system or the simplified model?
- Does “almost surely” require infinite time or unrealistic independence?
- Is there a counterexample in practical settings?

## Trap 11: Ignoring human factors

Formal methods fail socially when nobody understands or maintains the specs.

A coding agent can generate a 500-line proof that passes today. If no human can maintain it, the next change may delete it or route around it.

Mitigation:

- prefer readable specs,
- ask the agent to explain proof obligations,
- keep specs close to code,
- document why each invariant matters,
- train reviewers on the few formal constructs you use,
- start with small high-value properties.

## Trap 12: Forgetting that the agent can be attacked through data

Formal methods around tool policy must account for prompt injection and untrusted input. A web page, README, issue comment, or email can contain text that tells the agent to ignore instructions.

Formal policy should label untrusted data:

```text
web content = data, not instructions
issue comment = untrusted until reviewed
external docs = source material, not tool policy
```

Runtime monitors should enforce boundaries even if the LLM is convinced by malicious text.

This connects to AgentRaft’s data-flow concern and AgentGuard-style runtime checks: you cannot rely on the model to distinguish data from instructions perfectly.

## The skeptical checklist

Before trusting a formal-methods-plus-agent result, ask:

1. What exact property was checked?
2. Who reviewed that the property matches intent?
3. What assumptions does the model make?
4. Can the agent change the spec, tests, or protected code?
5. Are there examples showing the spec rejects bad cases?
6. Are runtime traces logged?
7. Does the proof/check cover code, workflow, data flow, or only a small helper?
8. What happens on checker timeout or unknown?
9. Is the source peer-reviewed, preprint, vendor, open-source, or community discussion?
10. What would convince us the result is wrong?

## The balanced conclusion

Formal methods are valuable precisely because LLMs are unreliable generators. But they do not remove the need for engineering judgment.

The right slogan is:

```text
LLMs lower the cost of trying formal methods.
Formal methods lower the risk of trusting LLMs.
Humans still own intent, assumptions, and risk acceptance.
```

If you remember only one warning, remember this: a green check is not a guarantee unless you know what the check means.
