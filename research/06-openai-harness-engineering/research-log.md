## Symphony Source Inspection

Local source snapshot:

```text
tmp/research-web-critical/openai-harness-engineering/symphony-source/
```

Inspected files:

- `README.md`
- `SPEC.md`
- `elixir/README.md`
- `elixir/WORKFLOW.md`
- `elixir/lib/symphony_elixir/agent_runner.ex`
- `elixir/lib/symphony_elixir/config/schema.ex`
- `elixir/lib/symphony_elixir/workspace.ex`
- `elixir/lib/symphony_elixir/orchestrator.ex` via `rg`
- `elixir/docs/logging.md`
- `elixir/docs/token_accounting.md`

Findings:

- Symphony is a long-running scheduler/runner for issue work, not a generic workflow engine.
- The spec centers polling Linear, selecting eligible issues, creating isolated workspaces, loading repo-owned `WORKFLOW.md`, running a coding agent, retrying, and exposing operator-visible observability.
- The Elixir README says it launches Codex in app-server mode and can expose a `linear_graphql` tool.
- The workflow file configures Linear states, workspace root, hooks, max concurrent agents, max turns, Codex command, sandbox, approval policy, and network access.
- Source files show workspaces are created per issue, hooks can run around workspace lifecycle, workspace paths are safety-checked, and terminal issues can trigger cleanup.
- Orchestrator search shows running, blocked, retry, reconciliation, stall, and terminal-state handling.
- Logging docs require issue IDs, issue identifiers, and session IDs for searchable lifecycle events.
- Token accounting docs explain live token usage and cumulative versus delta semantics for dashboards.

Boundary:

- Symphony is public OpenAI code, but it is not proof of the private harness implementation described in the blog. The guide labels Symphony-derived claims as public orchestration patterns or inference.
